# TRAE 用户规则（经典版）

## 目标与范围

- 明确约束与最佳实践，规避大模型常见编程问题
- 统一技术栈、接口风格、样式规范与工具使用流程
- 强化变更安全、可验证性与可回滚性

## 技术栈遵循

- 后端：Symfony（控制器、路由、FormType、Validator、Security）
- 模板：Twig（严格闭合 `block`、避免宏作用域滥用）
- ORM：Doctrine（遵循实体与仓库已有模型，禁止擅自新建同类表/模型）
- 前端：jQuery + 框架自研UI样式库
- 图标：Font Awesome 免费图标
- Ajax：使用框架封装的 `public/lib/ef/base/ajax.js`
- JSON 响应：统一使用 `src/Controller/Api/ApiResponse.php`

## 代码变更准则

- 变更前先阅读文件最新内容，避免覆盖他人改动
- 使用补丁方式进行修改；一次补丁仅修改一个文件
- 对未打开过的文件，先读取再修改；连续三次补丁必须重新读取确认
- 模仿项目现有代码风格、命名与目录结构；不擅自引入新库/新框架
- 任何涉及树结构的改动必须保持 Doctrine NestedSet 兼容
- Twig 模板必须确保所有 `block` 正确闭合，避免语法错误

## 工具使用准则

- 文件修改：优先使用补丁工具（避免 shell 直接写文件）
- 运行命令：仅在需要验证时使用；不要用来创建/编辑文件
- 搜索理解：优先用搜索代理做架构级检索，再用按文件读取细化
- 预览验证：Web 页面使用预览或 curl；无需启动开发服务器
- 多工具并行：批量读取/搜索时并行调用以提升效率

## 搜索与读取准则

- 先用高层语义搜索定位模块与架构，再精准读取关键文件
- 路由/控制器/模板/样式四个维度交叉确认页面来源与影响范围
- 对同名/相似文件谨慎区分（如 `admin/platform/*` 与 `test/*`）

## 安全与隐私

- 禁止记录、打印或提交任何敏感信息（令牌、Cookie、密钥）
- 不在仓库中存储机密；不把敏感数据写入日志
- 所有外部请求需说明用途与必要性，默认谨慎关闭

## UI 与样式

- 必须优先使用 `/templates/ui/` 中的组件与样式
- 不直接使用第三方UI库；不自定义样式类名
- 图标仅使用 Font Awesome 免费图标
- 树结构与表格示例参考 `/templates/test/` 目录

## 接口与响应

- Ajax 请求使用框架自研库；统一错误处理与重试策略
- 所有 JSON 响应统一封装为 `ApiResponse` 格式
- 后端接口必须进行参数校验与权限校验

## 验证与测试

- 页面验证：使用 curl 或内置预览方式验证 200/内容是否正常
- 交互验证：必要时添加轻量 JS 校验或日志；避免污染生产逻辑
- 样式验证：确保在窄屏与常规分辨率下不溢出、不遮挡
- 修改布局时优先通过容器 Flex 约束（如 `flex:1; min-width:0; overflow:auto`）

## 常见问题规避清单

- 不引用不存在的库/类/函数；先搜索确认后再用
- 不创建重复或平行实体（如已有组织实体 Company/Department/Position 等）
- 不破坏 NestedSet 的 childrenHierarchy 渲染逻辑
- Twig 语法严格，`{% endblock %}` 不得遗漏
- 不在模板中随意定义复杂宏，优先使用参数映射（如 `icon_map`）
- 不在 CSS 中写死页面宽度；使用弹性布局约束
- 不绕过框架 Ajax/ApiResponse，避免接口风格不一致
- 禁止使用 doctrine:schema:update 命令，必须通过迁移文件进行数据库变更

## 性能与可维护性

- 减少重复代码；抽取为复用组件或函数
- 样式与布局改动优先影响局部容器，避免全局污染
- 在 `admin/*` 与 `test/*` 场景中保持隔离，杜绝相互影响

## 文档与可追溯

- 仅在显式要求时创建文档文件；默认优先编辑已有文档
- 文档放置在 `/documents/` 目录；结构清晰、可执行（含访问路径）
- 在答复中引用 `file_path:line` 格式定位源代码

## 交互规范

- 默认采取行动并完成闭环；不要停留在方案阶段
- 当出现不确定性时，做出最合理假设并记录
- 提供简洁的验证结果与访问路径，避免泛泛描述

## 输出规范

- 遵循简洁、结构化的说明；必要时分节与要点列表
- 命令与路径使用反引号包裹；避免杂乱格式
- 语言与用户一致（默认中文）；不输出 ANSI 码

## 风险控制

- 大改动分步实施，每步可验证与可回滚
- 修改前后都进行读取与对比，确保预期一致
- 对潜在影响范围主动标注并避免牵连核心模块

## 示例建议（面向本框架）

- 组织树：通过 `childrenHierarchy` + `icon_map` 实现类型图标映射
- 表格页：`right-content` 使用 `flex:1; width:0; min-width:0; overflow:auto`，表格容器 `width:100% !important`
- Ajax：统一在 `ajax.js` 上封装请求与错误处理
- JSON：统一用 `ApiResponse` 包装返回值与状态码
