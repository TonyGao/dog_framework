员工管理
=======

OA的本质是人、权利、事务(工作)、时间、资源(文件、会议室、资金、数据等等)、视角(不同角色关注的视角不同)、秘密。
由这些员工共同组成了一套协作系统，这套系统的核心是人，围绕人员以上几个元素交相呼应，形成一套完备的系统。

员工管理作为OA系统的一个重要模块，可以挖掘的特性还很多，但由于历史原因、技术限制、HR在国内受重视程度
等诸多因素，实际上现存OA系统的人力资源并没有深入挖掘和探索，市面上也确实有一些人力专业系统，但整体取向上
其实是人力部门的业务系统。这种专业人力系统的取向，也恰恰说明了OA这个方向没有被重视的原因，如果只是入转调离、
考勤、薪资、福利、培训、员工关系、证照管理、花名册这些东西，那确实很难收到管理层和其他部门中层管理的重视，
于员工有用，但对于员工个人这种用途已经分散在了考勤模块、请假出差、工资条、通讯录这些终端功能或表单中。

绩效长久以来都是公司管理层重视的方向，但现存人力系统围绕绩效这个方向因为自身自成一体，并没有很好的融入
其他系统，造成数据来源偏手工、主观评判(360环评等)，作用流于表面，缺乏主营业务的专业性等等现象，因此
这些EHR并没有出现受到广泛企业认可的、真正意义上有价值的绩效模块。

对于OA重度使用的单位，这里说的“重度使用”特指业务普遍采用OA表单、固定模块留痕，能够形成有价值的数据汇总，
以此作为依据提供给中层或高层管理人员以真实可靠的人员工作绩效报表。在捷效办公这款产品中，我们曾经做过此类
尝试，虽然我个人认为是能够达成一些此类目的，但也发现了片面性和局限性。最典型的就是OA这种产品很难涉及专业
领域，仅适合与表单能够紧密联系的场景，另外汇总数据的目的如果用于体现工作量和绩效高低，那也只适合计件、累计销售额
并且在现实中以提成绩效和有明显正向激励措施的岗位，比如销售、计件工人等。

考虑以上可能的视角，我们可以从三种视角和多种视图模式来规划员工管理模块。

* 视角一，人力视角，这是最常规最常见的界面形式，多以树状部门和人员列表为管理基础，一些EHR会辅以人才库、面试、Offer阶段、入职操作、离职退休等行为、时段来规划界面。人员管理配以单元格按钮或者上方同意按钮进行常规的人员调动、人员编辑、离职办理等等操作。界面和形式虽然各有不同，但总体上万变不离其宗。
* 视角二，IT视角，这是出于用户管理、系统维护的角度考虑，一般人力缺乏OA用户管理的专业性，一些工作不得不由专业IT完成。但是IT的视图信息应该受限，一些OA系统管理员是上帝视角，HR十分烦感这种设计，明显IT获取过多信息未必是好事^_^
* 视角三，领导视角，这是从管理岗的角度来看人员，关注点是团队中的位置、工作内容、工作成果等等。

## 功能规划与开发计划

为了完美落地上述**三个视角**，我们将系统功能划分为三个独立的模块，实现数据互通但职能分离。

### 1. 视角落地与模块划分

| 视角 | 对应系统模块 | 菜单路径 | 核心职能 | 数据权限 |
| :--- | :--- | :--- | :--- | :--- |
| **视角一：人力 (HR)** | **员工管理** | `组织架构 -> 员工管理` | 档案维护、入转调离、合同管理 | 全局/辖区内员工档案（含敏感信息） |
| **视角二：IT (System)** | **用户管理** | `系统管理 -> 用户管理` | 账号安全、角色授权、登录日志 | 仅账号信息（User实体），**不含**薪资/家庭等隐私 |
| **视角三：领导 (Leader)** | **我的团队** | `个人工作台 -> 我的团队` | 团队概览、绩效查看、任务分配 | 仅限直属及下级部门成员 |

> **开发优先级**：优先实现 **视角一（员工管理）**，这是数据的基础。视角二复用现有系统功能并增强关联，视角三作为后续高级功能迭代。

### 2. [HR视角] 员工管理界面与交互

本模块将采用**左树右表**的经典布局，实现高效的人员信息检索与管理。

#### 2.1 整体布局

* **左侧：组织架构导航树**
  * **内容**：展示集团、公司、部门的层级结构。
  * **交互**：
    * 点击节点：右侧列表联动刷新，显示该节点及子节点下的所有员工。
    * 搜索框：支持快速检索部门名称。
    * 展开/收起：支持层级展开与折叠。
  * **组件**：复用框架现有的树形组件 (基于 Doctrine NestedSet 自研)。

* **右侧：员工花名册（DataGrid）**
  * **顶部工具栏**：
    * **搜索区**：姓名/工号/手机号模糊搜索、状态筛选（在职/离职/试用）、入职日期范围筛选。
    * **操作按钮**：办理入职（新增）、批量导入、导出Excel、批量调整部门、删除（逻辑删除）。
  * **数据表格**：
    * **核心字段**：工号、姓名、性别、所属公司、所属部门、主岗位、职务级别、员工状态（正式/试用/实习/离职）、入职日期、手机号。
    * **行操作**：查看详情、编辑、转正、调岗、离职。
  * **底部**：分页组件。

#### 2.2 详情页交互（Drawer/Modal）

* 采用 **侧滑抽屉 (Drawer)** 形式展示人员详情，便于在不离开列表的情况下快速查看和比对。
* **分栏展示**：
  * 基础信息（头像、工号、姓名、性别、生日等）
  * 职位信息（部门、岗位、汇报对象、职级、员工类型、状态）
  * 个人信息（证件类型/号码、联系方式、户籍地址、居住地址）
  * 合同信息（合同主体、期限、签订日期、到期日期）
  * 教育/工作经历（作为子表展示）

### 3. 功能模块细分

#### 3.1 基础信息管理

* **人员列表**：支持多维度筛选、列显示定制。
* **人员新增（办理入职）**：
  * 分步表单或长表单：包含账号信息、职位信息、个人信息。
  * **账号联动**：创建员工时，可选择自动创建关联的系统登录账号 (User实体)。
* **人员编辑**：支持对各项信息的修改。
* **人员查看**：只读视图，支持打印或导出个人档案。

#### 3.2 异动管理（Lifecycle）

* **转正**：调整员工状态字段，记录转正日期。
* **调动**：调整部门/岗位，生成调动记录（可选）。
* **离职**：填写离职原因、离职日期，系统自动禁用关联账号，将员工状态置为“离职”。
* **复职**：针对离职人员的重新入职流程，保留原有历史记录。

### 4. 数据模型设计 (Entity)

主要基于 `App\Entity\Organization\Employee` 实体，需确认包含以下关键属性：

| 字段分类 | 字段名 | 类型 | 说明 |
| :--- | :--- | :--- | :--- |
| **标识** | id | UUID | 主键 |
| | empNo | String | 工号 (唯一索引) |
| | user | Relation | 关联系统账号 (User OneToOne) |
| **基本** | name | String | 姓名 |
| | gender | Enum/String | 性别 |
| | mobile | String | 手机号 |
| | email | String | 企业邮箱 |
| | avatar | String | 头像URL |
| **职位** | company | Relation | 所属公司 (ManyToOne) |
| | department | Relation | 所属部门 (ManyToOne) |
| | position | Relation | 主岗位 (ManyToOne) |
| | level | Relation | 职级 (ManyToOne) |
| | type | String | 员工类型 (全职/兼职/实习/外包) |
| | status | String | 状态 (试用/正式/离职) |
| **时间** | entryDate | Date | 入职日期 |
| | regularDate| Date | 转正日期 |
| | leaveDate | Date | 离职日期 |
