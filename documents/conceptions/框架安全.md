# 企业级框架安全排查与漏洞修复整体方案

## 1. 概述

本文档旨在为企业级框架（基于 Symfony + Twig + jQuery/ES6 架构）的安全体系建立基线标准。文档涵盖前端、后端及框架层面的核心安全规范，并为接下来的代码漏洞扫描、排查及修复工作提供整体的执行方案和核对清单。

当前版本已经覆盖基础安全控制，但要做到真正可落地、可审计、可持续，还需要补足以下几个维度：威胁建模、接口级鉴权、输入输出边界、供应链治理、日志与审计、应急响应以及安全验收标准。本文档在原有基础上补充这些内容，目标不是“写一份原则清单”，而是形成一套可以直接指导代码扫描、修复实施和上线验收的安全方案。

## 1.1 安全目标

* **保密性**：防止账号、密码、Token、密钥、隐私数据、业务敏感字段泄露。
* **完整性**：防止越权修改、批量篡改、伪造请求、非法上传、配置被恶意变更。
* **可用性**：防止暴力破解、恶意刷接口、异常流量、任务阻塞导致系统不可用。
* **可审计性**：关键操作必须留痕，可追踪责任人、请求来源、时间和结果。

## 1.2 安全原则

* **默认拒绝**：任何未显式授权的访问一律拒绝。
* **最小权限**：用户、接口、数据库账号、文件目录、第三方凭证都只授予最小权限。
* **输入不可信**：所有来自用户、浏览器、第三方系统、消息队列、文件上传的数据都视为不可信。
* **输出可控**：输出到 HTML、JSON、日志、邮件模板、导出文件前必须经过编码、脱敏或过滤。
* **纵深防御**：前端校验不能代替后端校验；控制器校验不能代替服务层校验；业务鉴权不能代替基础设施鉴权。

## 2. 前端安全规范 (Front-end Security)

前端是系统面对攻击的第一道防线，主要关注防篡改与防伪造。

* **跨站脚本攻击 (XSS)**
  * **规范**：严禁在前端直接拼接用户输入生成 DOM。使用 Twig 渲染时，默认依赖框架的自动转义功能；富文本编辑器（如 WangEditor）的内容输出，必须在展示时进行 XSS 净化（如使用 `DOMPurify` 或 Symfony 的 `HtmlSanitizer`）。谨慎使用 Twig 的 `|raw` 过滤器。
  * **扫描/排查点**：全局搜索 `.twig` 文件中的 `|raw`，以及 `.js` 文件中的 `.html()`, `.append()`, `innerHTML` 等高危 DOM 操作方法。
* **跨站请求伪造 (CSRF)**
  * **规范**：所有状态变更的请求（POST, PUT, DELETE, PATCH）必须携带 CSRF Token。
  * **扫描/排查点**：检查前端 `/public/lib/ef/base/ajax.js` 中是否全局统一处理了 CSRF Token，以及 Symfony 表单是否禁用了 CSRF 保护（查找 `csrf_protection => false`）。
* **敏感信息防泄漏**
  * **规范**：禁止在 LocalStorage / SessionStorage / Cookie 中明文存储密码、Token、个人敏感信息（如完整手机号、身份证等）。
  * **扫描/排查点**：审查前端存储操作代码（`localStorage.setItem`），敏感数据应由后端脱敏后再返回前端。
* **点击劫持 (Clickjacking)**
  * **规范**：防止网站被恶意 iframe 嵌套，需配置 HTTP 响应头。
  * **扫描/排查点**：确认系统是否配置了 `X-Frame-Options: SAMEORIGIN` 或 `Content-Security-Policy: frame-ancestors 'self'`。
* **跨域资源共享 (CORS)**
  * **规范**：如果存在前后端分离或第三方 API 调用，严格限制 `Access-Control-Allow-Origin`，禁止使用通配符 `*` 并同时允许携带凭证 (`Access-Control-Allow-Credentials: true`)。
  * **扫描/排查点**：检查后端 CORS 配置（如 `nelmio_cors.yaml` 或自定义中间件），验证 Origin 校验逻辑。
* **内容安全策略 (CSP)**
  * **规范**：为后台系统、富文本页面、上传预览页面配置 `Content-Security-Policy`，尽量禁止内联脚本和未知来源脚本执行。
  * **扫描/排查点**：检查响应头是否存在 `Content-Security-Policy`，并确认 `script-src`、`frame-ancestors`、`object-src` 等配置不是宽松放行。
* **前端依赖与第三方资源**
  * **规范**：禁止直接使用未知来源脚本、样式、浏览器插件注入脚本。第三方前端库必须固定版本并纳入仓库或制品管理。
  * **扫描/排查点**：排查模板和基础布局中是否存在外部脚本地址、动态拼接脚本地址、未锁版本的第三方资源。
* **缓存与浏览器痕迹**
  * **规范**：包含隐私数据、后台页面、敏感查询结果的页面和接口必须控制浏览器缓存策略。
  * **扫描/排查点**：检查响应头是否合理设置 `Cache-Control`, `Pragma`, `Expires`，确认退出登录后敏感页面不可通过后退恢复。

## 3. 后端安全规范 (Back-end Security)

后端是系统的核心数据与逻辑堡垒，主要关注访问控制与数据完整性。

* **SQL 注入 (SQL Injection)**
  * **规范**：严禁直接拼接 SQL 字符串。必须使用 Doctrine ORM / DBAL 的 QueryBuilder，或者严格使用参数化查询（Prepared Statements）。
  * **扫描/排查点**：全局搜索代码库中的 `->executeQuery()`, `->executeStatement()`，以及任何尝试用 `.` 拼接 SQL 字符串的行为。
* **越权访问 (IDOR / Broken Access Control)**
  * **规范**：
    * **水平越权**：用户只能访问/修改属于自己的资源（操作前需校验数据的归属关系）。
    * **垂直越权**：低权限用户不能访问高权限（如管理员）的接口。
  * **扫描/排查点**：审计 Controller 层的实体获取逻辑。如果仅通过 `$id` 查询而不校验 `$entity->getOwner() === $currentUser`，或者接口缺少 `@IsGranted` 权限注解，则视为高危。
* **文件上传安全**
  * **规范**：必须在服务端校验文件的扩展名和 MIME 类型（使用白名单机制）。上传目录（如 `public/uploads/`）必须限制执行权限，防止 WebShell 上传。必须重命名上传文件（使用 UUID 或随机 Hash），禁止使用用户原始文件名。
  * **扫描/排查点**：检查文件上传相关的 Controller/Service，确保使用了 Symfony 的 `File` 约束和严格的 `mime_types` 校验，绝不能仅依赖前端校验。检查文件保存逻辑是否采用了随机重命名。
* **暴力破解与撞库**
  * **规范**：登录接口需限制失败次数（Rate Limiting）。密码重置等敏感操作应设置频率限制与有效期。
  * **扫描/排查点**：检查 `security.yaml` 中是否开启了登录防刷机制，以及验证码发送等接口是否有调用频率限制。
* **敏感数据加密存储**
  * **规范**：密码必须使用强哈希算法（如 Argon2i/bcrypt，Symfony 默认已支持）。数据库中的敏感隐私数据（如手机号、身份证号）建议加密存储。
  * **扫描/排查点**：检查 `security.yaml` 中的 `password_hashers` 配置。
* **批量赋值与过度发布 (Mass Assignment / Over-Posting)**
  * **规范**：禁止将 HTTP Request 数据直接全部绑定到 Entity 上。必须通过 Symfony Form 的字段白名单，或者手动获取特定字段进行赋值。
  * **扫描/排查点**：排查 API Controller 中是否有类似 `$entity->fromArray($request->request->all())` 这种不做字段过滤的危险赋值操作。
* **服务器端请求伪造 (SSRF)**
  * **规范**：凡是支持 URL 抓取、Webhook 回调、远程图片下载、第三方接口中转的功能，必须限制协议、域名、端口和跳转行为，禁止访问内网地址、元数据地址和本机敏感端口。
  * **扫描/排查点**：排查 `file_get_contents`, `curl`, `HttpClient`, Guzzle 等远程请求逻辑，确认是否校验目标地址，是否屏蔽 `127.0.0.1`、`localhost`、`169.254.169.254`、内网网段。
* **命令执行与反序列化**
  * **规范**：严禁将用户输入直接拼接到 `exec`, `shell_exec`, `proc_open`, `passthru` 等命令执行函数中；严禁反序列化不可信数据。
  * **扫描/排查点**：全局搜索命令执行函数、`unserialize()`、动态类实例化、动态文件包含等高危点。
* **业务幂等与重复提交**
  * **规范**：创建订单、发送验证码、发送邮件、审批、扣费、导入等关键写操作必须具备幂等控制或重复提交防护。
  * **扫描/排查点**：审计关键 Service 是否具备唯一请求号、去重键、状态机校验或事务控制。
* **事务与一致性**
  * **规范**：涉及多表修改、状态流转、库存/余额/权限变更时，必须使用事务并定义失败回滚策略。
  * **扫描/排查点**：检查 Service 层是否在关键写操作中使用 Doctrine 事务，是否避免在事务中执行高延迟外部调用。
* **日志与异常处理**
  * **规范**：禁止裸露异常堆栈给前端；错误日志中禁止记录密码、Token、验证码、完整身份证号、完整手机号、邮件密钥等敏感数据。
  * **扫描/排查点**：检查异常处理器、日志记录点和 API 响应，确认错误输出经过统一包装并且敏感字段已脱敏。

## 4. 框架层面安全 (Framework Security)

基于 Symfony 框架本身的基础设施和生态链安全。

* **依赖组件安全 (Supply Chain Security)**
  * **规范**：必须定期审计第三方 PHP (Composer) 和前端 (NPM) 依赖包的已知漏洞。
  * **扫描/排查点**：通过运行 `composer audit` 检查是否存在带有 CVE 漏洞编号的陈旧第三方包。
* **环境变量与敏感配置**
  * **规范**：`.env` 文件严禁提交到 Git 仓库（已在 `.gitignore` 中修复）。生产环境必须禁用 Debug 模式（`APP_DEBUG=0`），防止堆栈错误信息泄漏内部逻辑。
  * **扫描/排查点**：检查生产服务器部署配置，确认是否通过服务器环境变量或 Symfony Secrets 安全地管理了数据库密码、API Key 等敏感数据。检查 `.gitignore` 确保 `.env` 及敏感密钥未被提交。
* **会话与认证管理 (Session & Authentication)**
  * **规范**：Session Cookie 必须设置 `HttpOnly` 和 `Secure`（生产环境必须走 HTTPS），以及 `SameSite=Lax` 或 `Strict` 属性。
  * **扫描/排查点**：检查 `config/packages/framework.yaml` 中的 `session.cookie_secure` 和 `session.cookie_httponly` 等相关配置。
* **路由与控制器安全**
  * **规范**：严格区分内部 API（Admin使用）和外部开放 API。未授权不可访问的页面必须全局兜底拦截，避免遗漏。
  * **扫描/排查点**：排查 `config/packages/security.yaml` 中的 `access_control` 配置，确保默认路径 `/admin` 及其他核心路径处于严格保护状态。
* **HTTP 安全响应头**
  * **规范**：统一配置 `X-Content-Type-Options: nosniff`、`Referrer-Policy`、`Permissions-Policy`、`Strict-Transport-Security` 等基础安全响应头。
  * **扫描/排查点**：检查全局响应头中间件或 Web Server 配置，确认生产环境已开启 HSTS 且仅在 HTTPS 下生效。
* **部署与运行时安全**
  * **规范**：生产环境不得使用开发配置；调试工具、Profiler、测试路由、演示页面、测试账号必须在生产禁用。
  * **扫描/排查点**：排查 `/templates/test/`、测试控制器、Profiler、调试路由是否可能在生产暴露；检查 `APP_ENV`、`APP_DEBUG`、日志级别和错误展示策略。
* **数据库与存储权限**
  * **规范**：数据库账号按环境隔离，禁止使用超级管理员账号作为应用连接账号。上传目录、缓存目录、日志目录采用最小写权限。
  * **扫描/排查点**：检查部署配置、容器编排、数据库账号权限、`var/`、`public/uploads/` 等目录权限设置。
* **供应链与构建产物安全**
  * **规范**：锁定 Composer/NPM 版本，构建过程可重复，发布产物来源可追溯。禁止将临时脚本、备份文件、IDE 配置、测试产物纳入发布包。
  * **扫描/排查点**：检查 `composer.lock`、前端依赖锁文件、`.gitignore`、发布脚本和制品目录，确认无 `.bak`、测试脚本、备份目录进入仓库或制品。

## 5. 安全扫描范围与优先级

在实施扫描前，需要先划定范围，避免“全量扫描但没有重点”。建议按照以下优先级执行：

### 5.1 一级目标（必须优先完成）

* 认证登录、密码重置、Passkey、短信/邮件验证相关模块
* `/admin` 下的新增、编辑、删除、导入、审批、导出接口
* 文件上传、文件预览、富文本编辑、模板渲染、邮件预览
* 远程请求、Webhook、第三方登录、第三方通知回调
* 权限、菜单、角色、组织架构、员工信息等核心主数据

### 5.2 二级目标（完成一级后跟进）

* 查询接口的数据越权与批量导出
* 定时任务、命令行任务、初始化脚本、迁移脚本
* 缓存、日志、异常处理、消息通知、邮件发送服务
* 前端基础组件库、公共 Ajax 封装、模板基类布局

### 5.3 风险评级建议

* **高危**：可导致未授权访问、敏感数据泄漏、命令执行、SQL 注入、任意文件上传、权限提升、密钥泄漏
* **中危**：可导致业务数据被枚举、重复提交、弱校验绕过、日志泄漏、缓存泄漏
* **低危**：安全头缺失、提示信息过多、风控不完善、审计不完整

## 6. 漏洞排查与修复执行计划 (整体方案)

为确保系统安全无死角，接下来的扫描与修复工作将按照以下四个阶段执行：

### 阶段一：自动化基线扫描（Automated Scanning）

1. **依赖审计**：执行 `composer audit`，排查 Symfony 组件及第三方类库的已知漏洞。
2. **代码静态扫描**：结合自动化脚本/正则表达式，全局扫描高风险函数（如不安全的 SQL 拼接、未授权的路由、命令执行、反序列化、危险文件操作）。
3. **仓库卫生扫描**：检查 `.gitignore`、历史提交、临时文件、备份文件、测试文件、密钥和本地配置是否误入仓库。
4. **配置基线扫描**：检查 `security.yaml`、`framework.yaml`、Web Server 配置、环境变量配置、Cookie 策略和安全响应头。

### 阶段二：人工重点代码审计（Manual Code Review）

针对自动化工具难以发现的业务逻辑漏洞进行定向排查：

1. **审计目标 1（XSS 隐患）**：全局检索 Twig 模板中的 `|raw` 标记，逐一确认其变量来源是否 100% 可信，尤其是通过富文本编辑器录入的数据。
2. **审计目标 2（越权隐患）**：审计所有接收外部输入（`$request->query`, `$request->request`）并进行数据修改、删除的 Controller，确认是否做了完善的归属权校验。
3. **审计目标 3（SQL 注入隐患）**：人工排查 Repository 层中自定义的 DQL/SQL，确保所有外部变量都通过 `setParameter()` 绑定，而非字符串拼接。
4. **审计目标 4（服务层安全边界）**：检查业务规则是否只写在 Controller 中，如果 Service 可被多个入口调用，需确认 Service 层也具备权限和状态校验。
5. **审计目标 5（敏感输出）**：审计日志、导出、邮件模板、消息通知、API 响应，确认是否发生敏感字段明文输出。

### 阶段三：漏洞修复与防护加固（Remediation）

1. 汇总第一、二阶段的发现，输出《安全漏洞修复清单》。
2. 按照 **高、中、低** 优先级依次修复。对于高危漏洞（如 SQL 注入、严重的越权），需立即重构代码。
3. 在核心业务点（如登录、密码重置、越权敏感操作）补充和完善安全日志记录 (Audit Log)。
4. 对高风险模块补充自动化测试，至少覆盖鉴权、状态机、参数边界、非法输入、重复提交、上传校验。
5. 修复完成后进行回归验证，避免“修了一个漏洞，引入一个业务缺陷”。

### 阶段四：安全机制常态化（Process Integration）

1. 将 `composer audit` 等安全检查规范化，建议后续引入 CI/CD 流程中。
2. 建立代码 Review 规范：凡是涉及权限变更、资金/支付、文件上传的代码修改，必须重点 Review 安全边界。
3. 建立发布前安全检查清单，未通过不得上线。
4. 建立漏洞响应机制，包括发现、定级、修复、复测、发布、复盘六个步骤。

## 7. 发布前安全验收清单

以下检查项建议作为上线前的必核清单：

* 高危漏洞必须全部关闭，中危漏洞需有明确缓解措施和排期。
* 关键接口均已验证登录态、权限、归属权、参数合法性和重复提交防护。
* 上传、导入、富文本、模板渲染、导出功能已进行专项验证。
* 日志、报错、邮件、导出文件中不存在敏感数据明文泄漏。
* `.env`、密钥、证书、备份文件、IDE 文件、测试脚本未进入仓库与制品。
* 生产环境关闭 Debug、Profiler、测试路由和演示入口。
* 安全响应头、Cookie 属性、HTTPS/HSTS、CORS 策略均已验证。
* 所有新增修复已通过功能回归和接口测试。

## 8. 建议的交付物
