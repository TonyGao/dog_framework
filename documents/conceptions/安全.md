# 系统安全与三员管理开发计划书

## 1. 概述 (Overview)

本计划书旨在规划企业级框架的系统安全体系，重点实现**现代化 Passkey 无密码登录**以及符合**涉密/等保要求的“三员管理”原则**。通过权限分离与相互制约机制，构建高安全性的应用环境。

## 2. 身份认证升级：Passkey (FIDO2/WebAuthn)

为提升用户体验与安全性，系统将优先支持 Passkey（通行密钥）登录，逐步替代传统密码登录。

### 2.1 技术目标

- **无密码登录**：利用设备生物识别（TouchID, FaceID）或硬件密钥（YubiKey）进行认证。
- **抗钓鱼**：基于公钥加密体系，彻底解决撞库与钓鱼网站威胁。
- **跨设备同步**：支持 iCloud Keychain, Google Password Manager 等生态的密钥同步。

### 2.2 技术架构

- **协议标准**: FIDO2 / WebAuthn
- **后端依赖**:
  - `web-auth/webauthn-lib` (PHP 库)
  - 或 `spomky-labs/pwa-bundle` (Symfony 集成)
- **数据模型**:
  - `UserCredential`: 存储公钥 ID、公钥值、计数器、传输方式等。
  - 关联至 `Employee` 实体。

### 2.3 认证流程

1. **注册 (Registration)**:
   - 用户在已登录状态下（或通过初始临时密码），发起“绑定 Passkey”。
   - 服务器生成 `PublicKeyCredentialCreationOptions` (包含 Challenge)。
   - 浏览器调用 `navigator.credentials.create()`。
   - 用户验证生物特征，私钥保存在设备，公钥发送至服务器。
   - 服务器验证并存储公钥。
2. **登录 (Authentication)**:
   - 用户输入用户名（或点击“使用 Passkey 登录”）。
   - 服务器生成 `PublicKeyCredentialRequestOptions`。
   - 浏览器调用 `navigator.credentials.get()`，使用私钥签名 Challenge。
   - 服务器验证签名，颁发 JWT 或 Session。

## 3. 三员管理体系 (Three Safety Officers)

遵循《保密法》及分级保护要求，系统权限划分为三个相互独立、相互制约的角色，废除“超级管理员”(Root/SuperAdmin) 概念。

### 3.1 角色定义与职责

| 角色名称           | 英文标识         | 核心职责                                                                                                                            | 禁忌/互斥权限                                                                                |
| :----------------- | :--------------- | :---------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------- |
| **系统管理员**     | `ROLE_SYS_ADMIN` | **“管设备/管运行”**<br>1. 系统软硬件环境配置<br>2. 网络与存储配置<br>3. 系统升级与维护<br>4. 监控系统运行状态                       | **严禁**：<br>- 创建/修改用户账号<br>- 分配权限<br>- 查看业务数据<br>- 查看/删除安全审计日志 |
| **安全保密管理员** | `ROLE_SEC_ADMIN` | **“管人/管权限”**<br>1. 用户账号管理（开户、销户）<br>2. 角色与权限分配<br>3. 密级标识管理<br>4. 安全策略配置（密码策略、登录策略） | **严禁**：<br>- 系统运行配置<br>- 查看安全审计日志<br>- 访问业务敏感数据                     |
| **安全审计员**     | `ROLE_AUDITOR`   | **“管账/管监督”**<br>1. 查看系统日志与操作日志<br>2. 分析违规行为<br>3. 审计系统管理员与安全员的操作<br>4. 导出审计报表             | **严禁**：<br>- 任何配置修改操作<br>- 任何账号管理操作<br>- 具有“只读”属性                   |

### 3.2 权限控制实现 (RBAC)

- **Voters**: 实现自定义 Symfony Voter (`ThreeOfficersVoter`)，在 Controller/Service 层拦截互斥操作。
- **Menu Filtering**: 根据三员角色动态渲染左侧菜单，物理隔离不可见的功能入口。
- **初始化策略**: 系统初始化时，必须强制生成三个默认账号，分别对应三员，且**不可**合并为一人。

## 4. 系统安全管理功能

### 4.1 全面审计日志 (Audit Logging)

- **覆盖范围**:
  - 登录/登出（含失败尝试）
  - 三员的所有操作（增删改配置）
  - 业务数据的敏感操作（导出、批量删除）
- **日志属性**: `Who`(操作人), `When`(时间), `Where`(IP/设备), `What`(动作), `Result`(结果), `Old/New Value`(变更前后值)。
- **防篡改**: 审计日志存储于独立表或独立数据库，仅 `ROLE_AUDITOR` 可读，且**任何人不可删除/修改**（归档除外）。

### 4.2 密级管理 (Classification)

- **数据标记**: 核心业务实体（如文档、客户资料）需增加 `security_level` 字段。
- **访问控制**: 用户必须拥有 `>=` 数据密级的权限才可访问（Mandatory Access Control, MAC 思想）。

### 4.3 会话与防护

- **并发控制**: 限制同一账号同时在线数。
- **防暴力破解**: 连续失败锁定机制。
- **操作超时**: 无操作 N 分钟自动锁屏/登出。

## 5. 开发实施路线图 (Roadmap)

### 第一阶段：基础设施 (Infrastructure)

- [ ] 设计三员角色 (`SysAdmin`, `SecAdmin`, `Auditor`) 及对应权限集。
- [ ] 建立审计日志实体 (`AuditLog`) 与记录服务 (`AuditService`)。
- [ ] 改造现有 `Employee` 实体，剥离超级管理员权限。

### 第二阶段：Passkey 登录 (Authentication)

- [ ] 引入 `web-auth/webauthn-lib`。
- [ ] 创建 `UserCredential` 实体存储 Passkey 公钥。
- [ ] 开发前端 Passkey 注册与登录交互 (WebAuthn API)。
- [ ] 集成至 `SecurityController` 与 `AppCustomAuthenticator`。

### 第三阶段：三员功能实现 (Implementation)

- [ ] **系统管理员工作台**: 仅显示系统状态、缓存管理、路由配置等。
- [ ] **安全保密员工作台**: 用户管理（CRUD）、角色分配、部门管理。
- [ ] **审计员工作台**: 日志查询、高级筛选、报表导出。

### 第四阶段：加固与验收 (Hardening)

- [ ] 全局 Voter 校验，确保无越权漏洞。
- [ ] 审计日志不可篡改性测试。
- [ ] Passkey 兼容性测试（iOS, Android, Windows Hello, TouchID）。

## 6. 附录：Passkey 数据结构示例

```php
// Entity/Security/UserCredential.php
class UserCredential {
    private $id;

    #[ORM\ManyToOne(targetEntity: Employee::class)]
    private $user;

    #[ORM\Column(type: 'string', length: 255)]
    private $credentialId; // Base64URL encoded

    #[ORM\Column(type: 'text')]
    private $publicKey; // PEM or COSE

    #[ORM\Column(type: 'integer')]
    private $signCounter;

    #[ORM\Column(type: 'string')]
    private $transports; // usb, nfc, ble, internal

    #[ORM\Column(type: 'datetime')]
    private $createdAt;
}
```
