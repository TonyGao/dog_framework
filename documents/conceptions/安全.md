## 6. 密码重置功能 (Password Reset)

系统提供灵活的密码重置机制，允许管理员配置重置方式的优先级和启用状态。

### 6.1 重置方式配置 (Configuration)

在“安全配置”中维护重置密码方式列表，支持以下方式：

1. **Passkey (通行密钥)**: 使用绑定的 Passkey 进行身份验证后重置密码。
2. **Email (邮件验证)**: 发送重置链接或验证码到用户绑定的邮箱。
3. **SMS (短信验证)**: 发送验证码到用户绑定的手机号。
4. **Security Questions (安全问题)**: 回答预设的安全问题进行验证。

管理员可以：

- 启用/禁用特定方式。
- 调整方式的优先级（拖拽排序）。

### 6.2 重置流程 (Reset Flow)

用户点击“忘记密码”或“重置密码”时，系统将按照配置的优先级依次检查用户是否满足条件：

1. **获取配置**: 读取已启用且按优先级排序的重置方式列表。
2. **逐项检查**:
   - **Passkey**: 检查用户是否已绑定 Passkey。
   - **Email**: 检查用户是否绑定邮箱，且系统是否配置了发件服务。
   - **SMS**: 检查用户是否绑定手机号，且系统是否配置了短信网关。
   - **Questions**: 检查用户是否已设置安全问题答案。
3. **选择方式**:
   - 系统自动选择**第一个满足条件**的方式引导用户重置。
   - 若用户无法使用该方式（如手机丢失），可提供“尝试其他方式”选项（如果存在其他满足条件的方式）。
4. **执行重置**: 验证通过后，允许用户设置新密码。

### 6.3 实现状态 (Implementation Status)

- **配置管理**: 已实现 `/admin/security` 页面，支持 Passkey, Email, SMS, QA 方式的启用/禁用和优先级调整。
- **Passkey 重置**: 已实现。用户使用 WebAuthn 验证后可设置新密码。
- **Email 重置**: 已实现。系统发送验证码到用户邮箱（Mock），验证通过后可设置新密码。
- **SMS 重置**: 已实现。系统发送验证码到用户手机（Mock），验证通过后可设置新密码。
- **QA 重置**: 已实现。系统验证用户安全问题答案（Mock），验证通过后可设置新密码。
- **安全策略**: 重置密码后强制登出，要求用户使用新密码重新登录。

## 7. 认证与安全策略中心 (Auth & Security Policy Center)

在后台系统里做“密码强度配置”，本质上是把**安全策略参数化 + 可视化 + 可执行校验**。这个方向非常适合做成一个**可扩展的安全策略中心**，而不仅仅是一个简单表单。

以下从**产品设计 + 技术落地（Symfony方向）+ 可扩展性**三个层面的完整密码强度开发方案：

### 7.1 页面设计（Admin UI）

建议模块名：**安全中心 / Security Center → 密码策略**

#### 7.1.1 基础配置区（Password Policy）

**核心字段：**

- 最小长度（min_length）👉 默认 8
- 最大长度（max_length）👉 默认 32
- 必须包含：
  - [ ] 大写字母
  - [ ] 小写字母
  - [ ] 数字
  - [ ] 特殊字符
- 不允许包含：
  - 用户名
  - 邮箱前缀
  - 常见弱密码（字典校验）

👉 UI形式：checkbox + number input

#### 7.1.2 强度等级策略（Strength Rules）与 AI 智能配置

这里建议做成“规则组合”，而不是写死。为了避开复杂的 UI 配置（如繁琐的条件组拼接、拖拽等），我们可以**引入 AI 交互配置（AI Agent 辅助）**。

**设计思路：**
将复杂的低代码表达式 UI 简化为一个“自然语言对话框” + “可视化结果展示”的组合。这与系统已有的 `NaturalLanguageQueryAgent` 机制非常契合。

例如：

| 强度等级 | 自然语言配置指令示例 | AI 解析后的内部条件 (JSON DSL) |
| --- | --- | --- |
| 弱 | "长度大于等于6位就行" | `{"field": "length", "operator": ">=", "value": 6}` |
| 中 | "至少8位，而且必须包含数字" | `{"and": [{"field": "length", "operator": ">=", "value": 8}, {"field": "has_number", "operator": "=", "value": true}]}` |
| 强 | "至少10位，必须有数字和大小写字母" | `{"and": [{"field": "length", "operator": ">=", "value": 10}, {"field": "has_number", "operator": "=", "value": true}, {"field": "has_case", "operator": "=", "value": "mixed"}]}` |

👉 **AI 辅助 UI 交互流程：**

1. **自然语言输入**：管理员在输入框中输入：“把强密码规则设置为至少12位，必须包含大小写、数字和特殊符号，且不能包含用户名”。
2. **Agent 解析**：调用后端的 `PasswordPolicyAgent`，将自然语言转化为 JSON DSL 表达式。
3. **可视化确认**：前端收到 JSON 后，将其渲染成直观的标签（如：`[长度 ≥ 12]` `[含数字]` `[含大写]` `[含小写]` `[含特殊字符]` `[排除用户名]`）。
4. **微调与保存**：管理员如果觉得没问题，直接保存；如果有偏差，可以通过点击标签进行微调，或者继续对话补充条件（“再加一条：不能是最近用过的3个密码”）。

这样不仅极大降低了管理员的配置门槛，还凸显了企业框架的智能化（AI-Native）特性。

#### 7.1.3 密码生命周期（Password Lifecycle）

- 密码有效期（90天 / 永不过期）

- 是否允许重复历史密码
  - 记住最近 N 次密码
- 首次登录是否强制修改密码

#### 7.1.4 登录安全策略（附加）

（这个非常关键，很多系统忽略）

- 连续失败锁定：
  - 5 次失败 → 锁定 30 分钟
- 是否启用验证码
- 是否启用二次验证（2FA / Passkey）

#### 7.1.5 实时强度检测（前端体验）

👉 用户输入密码时：

- 显示强度条（弱 / 中 / 强）
- 实时提示：
  - ❌ 缺少特殊字符
  - ❌ 长度不足

👉 这部分建议统一做一个 JS 组件（类似 ajax.js 那种库）

### 7.2 数据结构设计（关键）

不要写死规则，一定要**可配置化 + 可扩展**

#### 7.2.1 表：`security_password_policy`

```sql
id
min_length
max_length
require_uppercase
require_lowercase
require_number
require_special
forbid_username
forbid_common_password

```

#### 7.2.2 表：`password_strength_rules`（核心扩展点🔥）

```sql
id
name           -- 弱 / 中 / 强
expression     -- JSON / DSL
sort_order
```

示例 expression：

```json
{
  "min_length": 10,
  "require": ["number", "uppercase", "special"]
}
```

👉 或者更高级（推荐使用）：

```json
{
  "and": [
    {"field": "length", "operator": ">=", "value": 10},
    {"field": "has_number", "operator": "=", "value": true},
    {"field": "has_special", "operator": "=", "value": true}
  ]
}
```

👉 这跟 datagrid 的过滤设计可以复用一套引擎！
